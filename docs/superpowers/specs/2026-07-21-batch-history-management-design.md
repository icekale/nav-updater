# 更新历史批量管理设计

日期：2026-07-21

状态：待用户复核

## 目标

在“更新历史”中支持研究员选择当前页的多个批次，统一执行批量重新生成或永久删除，减少逐条操作的重复点击，同时保持单批次操作已有的安全边界和审计记录。

## 范围

首版仅管理当前历史页已加载的最多 50 条批次，不增加跨页选择、筛选、归档、批量下载或账号批量管理。

可用操作：

1. 批量重新生成结果；
2. 批量永久删除。

## 交互

历史表第一列新增复选框，表头提供“全选当前页”。未选中批次时，批量操作栏不显示；选中后，在表格上方显示“已选 X 条”、`重新生成结果` 与低强调危险操作 `永久删除`。

点击“重新生成结果”立即提交当前勾选项；点击“永久删除”使用浏览器原生确认，明确说明会永久清理所选批次的上传文件、结果文件和关联记录。

单条“查看”和“删除”入口保留，不改变现有工作流。

## 服务端规则

新增一个受 CSRF 保护的批量操作路由。请求包含 `run_ids` 列表和 `action`，只接受 `requeue` 与 `delete` 两个动作；空选中、未知动作或无有效批次返回可见错误。

每个选中批次独立处理：

- 重新生成：非 `processing` 批次恢复到队列，写入原有 `queue/update_run` 审计；`processing` 批次跳过；
- 删除：复用 `delete_run`，因此仍永久删除 RunFile、RunItem、批次目录、结果 Excel 与相关批次审计；`processing` 批次跳过；
- 单个批次的异常不会撤销已经成功处理的其他批次；路由汇总成功、跳过和不存在的数量，并重定向回历史页显示通知；
- 删除和重新生成均继续以“每个批次一条审计记录”保存，不写一份含有指标数据的批量审计明细。

批量删除不绕过现有路径安全限制；数据目录外的错误路径仍不会被删除。

## 结果反馈

通知文案分别汇总重新生成数、删除数、处理中跳过数和不存在数。例如：“已重新生成 3 个批次，跳过处理中 1 个”。当所有选中批次都无法执行时，仍显示准确的跳过或不存在数量。

## 测试与验收

1. 历史页可选择单条、全选当前页和取消全选；未选中时不展示批量操作栏。
2. 批量重新生成两个已完成批次后，两者都处于 `uploaded`，并各有一条 `queue/update_run` 审计。
3. 批量删除两个已完成批次后，其文件、子记录和旧审计均被删除；每个批次各有一条不含指标明细的删除审计。
4. 选中集内有 `processing` 批次时，它保持存在且不重排队、不删除；其他有效批次仍成功处理。
5. 空选中、未知动作和不存在批次均不会影响其他批次，并给出可见反馈。
6. 全量 pytest 和 Ruff 继续通过；在 Unraid 部署后健康检查正常。

## 非目标

- 不新增跨页全选、筛选、回收站或撤销操作；
- 不改变单条删除、单条重新生成或 OCR 处理规则；
- 不增加产品、账号或会议记录的批量操作。
