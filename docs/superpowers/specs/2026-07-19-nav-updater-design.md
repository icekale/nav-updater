# 投研净值更新工具设计文档

日期：2026-07-19

状态：待用户书面复核

范围：第一阶段 MVP，仅包含净值 Excel 更新工具

## 1. 目标

在同一局域网内提供一个支持多人访问的网页工具。用户上传原始净值 Excel 和一张或多张高清净值截图后，系统自动识别私募/渠道产品数据、查询缺失的公募数据、计算约定指标、生成保持原结构的新 Excel，并保存完整处理历史。

第一阶段不实现资本市场会议采集、AI 分析、人工审核或微信提醒。这些能力属于第二阶段，但第一阶段保留统一的用户与审计基础，避免后续重复建设。

## 2. 已确认约束

- 部署在局域网内一台长期在线且已安装 Docker 的主机上。
- 多名内部用户通过浏览器访问，不开放公网。
- 使用 Docker Compose 快速部署。
- 私募/渠道产品数据由用户人工上传高清截图。
- 公募基金净值由系统在用户发起处理批次时自动获取；第一阶段不包含无人值守定时任务。
- 用户会提供高清净值样例和产品代码清单。
- 原始 Excel 中产品名称为空的行保持不变。
- 计算使用累计净值。
- 匹配失败或数据不足时保留上次值，并将相关单元格标红。
- 每次处理都保存输入、结果、操作人和处理状态。

## 3. MVP 用户流程

1. 用户使用内部账号登录。
2. 用户进入“新建更新”页面，上传一份 `.xlsx` 文件和一张或多张 PNG/JPEG 截图，并确认截止日；截止日默认取运行日期之前或当天最近一个周五。
3. 系统校验 Excel 表头、文件类型和文件大小。
4. 系统对截图执行 OCR，提取产品名称、产品代码和可识别的绩效字段。
5. 系统将截图记录与 Excel 产品行匹配；未在截图中匹配且有公募代码的产品进入公开数据查询流程。
6. 系统生成处理预览，列出成功匹配、待确认、数据不足和查询失败的产品。对于待确认项，用户可以修正 OCR 值、选择目标产品或跳过该项。
7. 用户确认后，系统计算或采用已提取的指标并生成新 Excel；仍未解决或被跳过的行保留旧值并标红。
8. 用户下载结果文件，也可在“更新历史”中重新下载以往输入和输出。

系统永远生成新文件，不覆盖用户上传的原文件。

## 4. 技术架构

### 4.1 Docker Compose 服务

第一阶段包含三个长期运行的服务：

- `app`：Python 3.12 + FastAPI，负责页面、登录、文件上传、预览、确认和下载。
- `worker`：与 `app` 使用相同镜像，独立进程执行 OCR、产品匹配、公开数据查询、指标计算和 Excel 输出。
- `db`：PostgreSQL 16，保存用户、产品、净值观测值、处理批次、逐行结果和文件元数据。

上传文件与输出文件保存在 Docker 持久卷，不写入容器临时层。第一阶段不引入 Redis 或对象存储；PostgreSQL 中的批次记录同时作为持久任务队列。`worker` 使用事务和 `FOR UPDATE SKIP LOCKED` 原子认领一个待处理批次，CPU 密集型 OCR 不占用网页请求进程。`worker` 异常重启后，将超过 30 分钟且无心跳的运行中批次重新置为待处理；同一批次每次只有一个 worker 可以认领。

### 4.2 访问方式

- 应用默认监听 `0.0.0.0:8080`。
- 局域网用户通过 `http://<部署主机IP>:8080` 访问。
- Docker 服务设置健康检查和 `restart: unless-stopped`。
- `.env` 提供数据库密码、初始管理员账号、会话密钥和公开数据源所需配置。
- 首次启动自动执行数据库迁移并创建初始管理员。

### 4.3 前端范围

采用 FastAPI 服务端模板和轻量渐进增强交互，不单独建设 SPA。第一阶段页面仅包括：

- 登录页
- 新建更新页
- 识别与匹配预览页
- 处理结果页
- 更新历史页
- 产品代码维护页
- 账号管理页（仅管理员）

不实现仪表盘、复杂图表、手机端专门适配或自助注册。

## 5. 核心模块

### 5.1 文件接收与校验

- Excel 仅接受 `.xlsx`。
- 图片仅接受 PNG/JPEG。
- 服务端通过真实文件签名验证类型，不只依赖扩展名。
- 文件名随机化保存，原始文件名仅作为元数据记录。
- 单个文件默认上限 20 MB，单次最多 10 张图片。
- Excel 必须包含产品名称列及目标指标列；缺少必要表头时整批停止，不生成结果。

### 5.2 Excel 适配器

首个适配器面向已提供的《净值更新表.xlsx》：

- 产品名称位于 E 列。
- F-Q 分别为近一周、MTD、YTD、2019-2025 年收益、近一年夏普比和近一年最大回撤。
- 产品名称为空的行不读取、不查询、不写入。
- 保留工作表名称、合并单元格、行高、列宽、字体、边框和填充样式。
- 百分比指标在内部使用小数表示；写入模板时乘以 100 并四舍五入到两位小数，因为模板表头已经标明单位为 `%`。夏普比直接写入并四舍五入到两位小数。
- 第一阶段只填写模板中已经存在的年份列，不自动新增年份列。

为避免 `openpyxl` 保存时删除模板中的未知扩展、条件格式或其他包部件，输出适配器通过 XLSX ZIP/XML 接口只修改目标单元格及必要的新增红色错误样式。除目标工作表 XML 和样式表中追加的错误样式外，其他包部件保持字节不变。业务计算不依赖具体列号，方便以后增加其他模板而不改计算核心。

### 5.3 OCR 与截图表格解析

- 默认使用 CPU 可运行的 RapidOCR/ONNX Runtime 识别中英文和数字。
- 图像预处理包括方向纠正、缩放、对比度增强和必要的表格线检测。
- 表格解析根据文字坐标重建行列，不依赖纯文本顺序。
- OCR 结果保留原文字、标准化文字、坐标和置信度，便于审计。
- 产品名称、产品代码或关键数值低于置信度阈值时，不自动写入，状态设为“待确认”。置信度阈值在真实样例验收后固定在配置中，不由普通用户调整。
- 数值解析统一处理千位分隔符、百分号、负号和括号负数；无法唯一解释的值进入“待确认”。
- 截图中识别到的报告日期必须与批次截止日一致；不一致时必须由用户确认后才能继续。
- 截图明确包含的绩效字段视为该私募/渠道产品的权威输入，直接采用并记录来源为截图，不用公开净值重新计算。
- 本设计第 6 节的公式只用于公募历史净值，或截图确实提供了带日期的累计净值序列时的缺失字段计算。
- 若同一字段同时存在截图值和可计算值，预览页同时展示二者，默认采用截图值；用户可以在确认时改用计算值。

OCR 引擎封装在适配器后，若真实样例显示 RapidOCR 无法达到验收准确率，可替换引擎而不改变匹配、计算和 Excel 模块。

### 5.4 产品识别与匹配

匹配优先级如下：

1. 产品代码完全一致。
2. 标准化产品名称完全一致。
3. 未匹配且产品代码在产品清单中标记为公募时，按代码查询公开数据源。

名称标准化只处理空白、全半角符号、常见括号和已登记的历史名称，不进行模糊自动写入。相似名称只能进入“待确认”，避免把不同份额或同系列产品错误合并。

若 Excel 行在截图中找不到，系统按已确认规则尝试视为公募，但自动查询仍以产品代码为准；没有代码时不得仅凭名称猜测公募代码。

产品代码清单接受 `.xlsx` 或 `.csv`，必须包含三列：`product_name`、`product_code`、`product_type`。`product_type` 只允许 `public` 或 `private`。管理员在“产品代码维护”页导入，系统按代码去重并拒绝名称、代码或份额类别冲突；冲突必须人工修正后才能启用该产品。

### 5.5 公募数据提供器

- 公开数据访问封装为 `PublicFundProvider` 接口。
- 第一阶段使用东方财富公开基金净值接口作为默认提供器，按基金代码获取累计净值历史。
- 每个净值观测值保存产品代码、净值日期、累计净值、来源 URL、获取时间和原始响应摘要。
- 相同产品、日期和来源的数据使用唯一约束，重复运行不会生成重复记录。
- 网络超时、限流、返回结构变化或代码不存在时，当前产品失败但不终止整批处理。

公开接口可能发生变化，因此业务逻辑不得直接依赖特定站点的响应格式。若默认来源不可用，只替换提供器实现。

## 6. 指标口径

统一定义截止日 `T` 为用户提交批次时确认的日期，默认值是运行日期之前或当天最近一个周五。`T` 按 `Asia/Shanghai` 保存且在批次提交后不可变。每个指标同时记录实际采用的分子、分母净值日期。

某个产品在目标日期没有净值时，使用目标日期当日或之前最近一个有效净值，但禁止无限回退：公募有效净值距目标日期超过 14 个自然日即视为陈旧；私募累计净值或截图报告日期距 `T` 超过 45 个自然日即视为陈旧。陈旧数据不参与新指标计算，对应旧值保留并标红。

### 6.1 收益率

- 近一周：`T` 对应有效净值 / 上一个周五对应有效净值 - 1。
- MTD：`T` 对应有效净值 / 上月末对应有效净值 - 1。
- YTD：`T` 对应有效净值 / 上年末对应有效净值 - 1。
- 年度收益：当年末对应有效净值 / 上年末对应有效净值 - 1。

“对应有效净值”均指目标日期当日或之前最近一个有效累计净值。若分母不存在，则该指标状态为数据不足。

### 6.2 近一年夏普比率

- 观察区间为 `T` 往前一年至 `T`。
- 无风险利率按 0。
- 取区间开始日前最后一个有效净值作为首个收益率的基准；同日重复值保留最新一次成功导入的记录，非正净值视为无效。
- 公募使用相邻日频有效净值的简单收益率，年化因子为 252。
- 私募仅在系统拥有带日期的累计净值序列时计算；使用实际披露序列，年化因子为 `365.25 / 相邻观测日间隔中位数`，最高不超过 252。
- 公式为 `平均期收益 / 样本标准差 × sqrt(年化因子)`，样本标准差使用 `ddof=1`。
- 有效收益少于 3 个或标准差为 0 时，结果为数据不足。
- 数据库保留完整精度，页面和 Excel 仅在展示时四舍五入到两位小数。

### 6.3 近一年最大回撤

- 使用 `T` 往前一年至 `T` 的全部有效累计净值。
- 每个时点回撤为 `当前净值 / 历史峰值 - 1`。
- 最大回撤为区间内最小回撤值。
- 有效净值少于 2 个时，结果为数据不足。

## 7. 数据模型

第一阶段包含以下核心实体：

- `users`：账号、密码哈希、角色、启用状态、创建时间。
- `products`：产品代码、标准名称、产品类型、公募标记、历史名称。
- `nav_observations`：产品、净值日期、累计净值、来源、来源文件或 URL、获取时间。
- `update_runs`：操作人、截止日、状态、开始/结束时间、汇总数量。
- `run_files`：批次、文件类型、原始名称、存储路径、SHA-256。
- `run_items`：批次、Excel 行号、产品、匹配来源、行状态、各项指标值、逐指标状态和错误原因。
- `audit_logs`：操作人、动作、对象类型、对象 ID、时间和必要的非敏感上下文。

角色第一阶段只区分管理员和普通用户。管理员可维护产品代码和账号；普通用户可上传、处理、查看和下载所有历史批次。第一阶段按团队共享数据设计，不隔离不同用户的批次。

## 8. 状态与异常处理

处理批次状态为：`uploaded`、`parsing`、`needs_review`、`processing`、`completed`、`completed_with_warnings`、`failed`。

逐行结果分开记录三个维度，避免来源与质量状态互相覆盖：

- 匹配来源：`image`、`public_provider`、`none`。
- 行状态：`ready`、`needs_review`、`stale`、`failed`。
- 每个指标状态：`extracted`、`calculated`、`stale`、`insufficient_data`、`failed`。

产品名称为空的 Excel 行不创建逐行记录。

处理规则：

- Excel 结构错误：整批失败，展示可操作的错误信息。
- 单张图片解析失败：其余图片继续处理，批次进入警告状态。
- 产品无法匹配：保留原单元格值并标红，记录原因。
- 公募查询失败：保留原单元格值并标红，记录来源错误。
- 指标数据不足：只标红对应指标单元格，其他可计算指标正常写入。
- 输出文件先写入临时路径，校验可重新打开后再原子移动到正式路径。
- 写入顺序为：生成临时输出文件并验证可打开，原子移动到正式路径，最后在数据库事务中登记输出文件并完成批次。启动时对账文件与数据库记录，缺失或未登记的输出标记为失败。失败任务可重新运行，但不覆盖原批次记录。

## 9. 安全与审计

- 密码使用 Argon2id 哈希，不保存明文。
- 登录使用服务端会话 Cookie；Cookie 设置 `HttpOnly` 和 `SameSite=Lax`。
- 所有修改状态的表单请求使用 CSRF 令牌。
- 第一阶段不提供公网 HTTPS，部署说明明确要求仅在可信局域网使用。
- 所有上传、处理、下载、产品维护和账号维护操作写入审计日志。
- 输出文件只允许已登录用户通过应用下载，不直接暴露宿主机目录。
- Excel 读取禁止执行宏；第一阶段不接受 `.xlsm`。

## 10. 数据持久化与运维

- PostgreSQL 数据、上传文件和输出文件分别挂载持久卷。
- 容器重启或升级不得丢失历史。
- 部署包提供 `.env.example`、`docker-compose.yml`、初始化命令和健康检查说明。
- 提供数据库与文件卷的备份、恢复命令；备份由部署管理员手动执行。
- 应用日志输出到标准输出，包含请求 ID 和批次 ID，但不记录密码、会话或完整外部响应。

## 11. 测试策略

### 11.1 单元测试

- 周收益、MTD、YTD、年度收益的日期边界和缺失日期回退。
- 截止日冻结、公募 14 天和私募 45 天陈旧阈值。
- 日频及不规则频率夏普比率，包括样本标准差、重复日期、非正净值和舍入。
- 最大回撤。
- 名称标准化、代码匹配和拒绝模糊匹配。
- 产品代码清单导入、去重和冲突拒绝。
- Excel 行映射、空产品行跳过、失败标红和 XLSX 非目标包部件保持不变。

### 11.2 集成测试

- 使用用户提供的脱敏真实截图建立固定样例集。样例中所有被系统自动接受的产品代码、名称和数值字段必须与人工标注完全一致；不能确定的字段必须进入待确认，禁止静默写入错误值。
- 使用模拟公开数据响应验证正常、超时、限流和结构异常。
- 使用真实模板生成输出，通过 Excel/WPS 和独立解析器重新打开验证结构和数值，并逐个比较非目标 XLSX 包部件的哈希。
- 验证处理历史、文件哈希和数据库唯一约束。

### 11.3 端到端与部署测试

- 登录、上传、预览、确认、处理、下载的完整浏览器流程。
- 两名用户同时发起不同批次时互不覆盖。
- `worker` 处理中断后，超时批次可被安全重新认领且不会生成两个正式输出。
- `docker compose up -d` 后健康检查通过。
- 重启容器后历史批次和文件仍可访问。

## 12. 验收标准

第一阶段满足以下全部条件才算完成：

1. 在一台安装 Docker 的局域网主机上，可通过文档中的命令完成部署。
2. 其他局域网电脑可使用账号登录并访问应用。
3. 管理员可导入约定三列的产品代码清单，冲突数据不会被静默覆盖。
4. 上传真实样例 Excel 和截图后，可预览产品匹配结果，并可修正 OCR 值、选择产品或跳过待确认项。
5. 固定样例集中所有自动接受的截图字段与人工标注完全一致，不能确定的字段全部进入待确认。
6. 截图中能够明确识别的产品和指标正确写入对应 Excel 行。
7. 有基金代码的公募产品可自动获取净值并按本设计计算指标。
8. 产品名称为空的行完全不变。
9. 匹配失败、查询失败、陈旧或数据不足时保留旧值并标红，且页面可见原因。
10. 输出 Excel 可由 Excel/WPS 正常打开；除目标单元格和追加的错误样式外，其他 XLSX 包部件保持不变。
11. 历史页面可查看操作人、时间、处理状态并重新下载输入输出。
12. 容器重启后数据库和文件仍然存在，中断批次可安全恢复。

## 13. 明确不在第一阶段的内容

- 自动登录私募/渠道网站或绕过访问限制抓取数据。
- 会议、政策和公告采集。
- AI 摘要、投研映射与人工审核页面。
- 微信、企业微信或邮件提醒。
- 公网部署、HTTPS、单点登录和复杂权限系统。
- 移动端专门界面。
- 自动新增 Excel 年份列或支持任意未知 Excel 模板。

第二阶段另行设计，不在第一阶段提前创建无实际用途的模块或数据库表。
